Hazelcast 기반 Reverb 서버 구축

Hazelcast(IMDG) 사용해 Reverb(DeepMind의 분산 Replay Buffer) 역할을 하는 서버를 구축하는 방법을 “아키텍처 → 데이터 구조 → 샘플링 → 우선순위 → API 설계 → 예제 코드” 순으로, 아래 구조는 실제 Reverb가 하는 기능을 Hazelcast 기반으로 재현한 설계.

🧠 목표

Hazelcast를 이용하여 다음 기능을 수행하는 Reverb-like Distributed Replay Buffer Server 구축:

  • Actor → Replay 서버로 전이(transition) insert
  • Learner → 샘플링(pull)
  • Prioritized Experience Replay(PER) 지원
  • Sequence/trajectory 저장
  • 자동 eviction (FIFO 또는 priority-based)
  • 병렬 Actor 지원
  • 분산 노드 장애 대응(Hazelcast가 제공)

🏗️ 전체 아키텍처

Hazelcast 노드 수가 증가하면
→ replay buffer 용량, 처리량, fault-tolerance가 수평 확장됩니다.

🧱 핵심 구성 요소별 설계

1) Transition 구조 (Replay Item)

Reverb의 Item(trajectory/transition)을 Hazelcast에서 고정 포맷 JSON 혹은 Serializable 객체로 저장합니다.

2) 저장소 구조 (Hazelcast 데이터 모델)

📌 a) 메인 Replay Buffer: IMap<Long, Transition>

  • Key: 전역 증가 ID (AtomicLong 사용)
  • Value: Transition
  • Hazelcast가 자동 파티셔닝함 → 분산 buffer 완성

📌 b) Priority Buffer: IMap<Long, Float>

  • Prioritized Experience Replay(PER) 구현용
  • key: transition ID
  • value: priority 값

📌 c) Sampling Index

랜덤 또는 priority 기반 샘플링을 위해

  • 전체 key list를 캐시 or
  • partition-local index 유지 or
  • Hazelcast SQL로 랜덤 샘플링 가능

📌 d) AtomicLong: 글로벌 ID 생성기

idGenerator.getAndIncrement()

📌 e) Optional: Ringbuffer (trajectory/sequence replay)

Reverb의 Trajectory 기능을 위해:

  • Ringbuffer<Transition[]>
  • 또는 MultiMap<episode_id, Transition>

⚙️ API 설계

Actor → 서버

insert(transition, priority)

  • id = atomicLong.getAndIncrement()
  • transitionsMap.set(id, transition)
  • priorityMap.set(id, priority)

Learner → 서버

sample(batch_size)

  • uniform sampling → 랜덤 key 선택
  • prioritized sampling → priority 기반 확률 샘플링

update_priority(id, new_priority)

  • priorityMap.set(id, new_priority)

size()

  • transitionsMap.size()

🔥 Prioritized Experience Replay 구현

1) priorityMap 에 모든 우선순위 저장

2) 샘플링

샘플링 확률
$$ P(i) = p_i^\alpha / \sum_k p_k^\alpha $$

구현 방법 2가지:

✔ 방법 A: Hazelcast ExecutorService + partition local reduction

각 파티션에서 priority sum을 계산 → 합산
그 후 확률 기반으로 key 선택

✔ 방법 B: Hazelcast SQL 사용

Hazelcast 5.0+는 SQL 쿼리를 지원

→ 상위 PRIORITY N개를 가져와 확률 기반 샘플링
대규모 환경에서 매우 빠름

✔ 방법 C: Segment Tree (SumTree) 구현

PER의 정석

  • Hazelcast IMap에 segment tree 노드를 저장
  • 노드는 클러스터에 분산 저장 가능
  • priority 업데이트 빠름

추천 방식: A 또는 B – 구현 난이도 낮음 + Hazelcast의 분산 성능 활용

🧨 Hazelcast 기반 Reverb 서버의 장점

항목 Reverb Hazelcast 기반 Reverb
RL 특화 기능 내장 구현 필요
대규모 분산 매우 강함 매우 강함

Hazelcast는 “Replay Buffer + PER + Sequence storage + Sampling 서버”를 구현하기 위한 강력한 분산 인프라를 제공하지만 Reverb의 RL-specific 기능은 직접 설계해야 한다는 차이가 있음.

🎯 결론

Hazelcast로 충분히 Reverb와 동일한 역할을 하는 분산 Replay Buffer Server를 구축할 수 있습니다.

  • IMap 기반의 분산 replay storage
  • priorityMap 기반 PER
  • SQL/ExecutorService 기반 sampling
  • Ringbuffer 기반 trajectory
  • AtomicLong 기반 global id
  • EntryProcessor 기반 atomic priority update
  • Hazelcast cluster 확장성으로 Actor/Learner 대규모 운영 가능

완전한 Reverb 클론이 되고 싶다면 다음 모듈도 구현해야 합니다:

  • RateLimiter (insert:sample 비율 관리)
  • Trajectory assembler (sequence replay)
  • Table abstraction (여러 replay table 사용)
  • 배치 기반 샘플링 API

Spring Cloud Gateway vs Netty

“이미 완성된 고성능 프레임워크를 사용할 것인가, 아니면 핵심 라이브러리를 사용해 직접 만들 것인가” 라는 중요한 아키텍처 선택의 문제입니다.

Spring Cloud Gateway는 내부적으로 Netty를 사용하여 만들어졌습니다. 즉, Netty의 고성능 비동기 네트워킹 능력을 기반으로, 게이트웨이에 필요한 수많은 기능(라우팅, 필터링, 보안 등)을 추상화하여 미리 구현해 놓은 프레임워크 입니다. 반면 Netty는 이러한 프레임워크를 만들 수 있는 네트워킹 라이브러리 입니다.

두 가지 방식을 비교하면 다음과 같습니다.

비교 분석: Spring Cloud Gateway vs. 순수 Netty 직접 구현

항목 Spring Cloud Gateway (SCG) 순수 Netty로 직접 구현
추상화 수준 높음 (High-Level) 낮음 (Low-Level)
개발 속도 매우 빠름 (몇 시간 내에 기본 게이트웨이 구축 가능) 매우 느림 (수개월 이상 소요될 수 있음)
성능 매우 높음 (Netty 기반으로 이미 최적화됨) 이론상 최고 성능 가능 (하지만 전문가 수준의 최적화 필요)
기능 All-in-One (모든 기능 내장) 모든 것을 직접 구현해야 함
유연성 프레임워크의 규칙 안에서 유연함 완벽한 제어 및 최고의 유연성
복잡도 낮음 ~ 중간 (Spring 및 반응형 프로그래밍 이해 필요) 매우 높음 (TCP/IP, Netty 내부 동작 등 깊은 이해 필요)
유지보수 용이함 (Spring 생태계와 일관성 유지) 매우 어려움 (코드가 복잡하고, 담당자 의존성 높음)

1. Spring Cloud Gateway 사용 시 장점과 단점

✅ 장점

  1. 압도적인 생산성: 게이트웨이의 핵심 기능인 동적 라우팅, Predicate(조건부 라우팅), 필터(전/후처리) 등을 application.yml 설정과 간단한 Java 코드로 빠르게 구현할 수 있습니다.
  2. Spring 생태계와의 완벽한 통합:
    • Service Discovery: Eureka, Consul 등과 연동하여 마이크로서비스의 주소를 동적으로 찾아 라우팅(uri: lb://SERVICE-NAME)합니다.
    • Configuration: Spring Cloud Config를 통해 게이트웨이 설정을 중앙에서 관리하고 동적으로 변경할 수 있습니다.
    • Circuit Breaker: Resilience4j와 쉽게 통합하여 다운스트림 서비스의 장애가 전파되는 것을 막습니다.
    • Security: Spring Security와 통합하여 인증/인가(OAuth2, JWT 등)를 손쉽게 구현할 수 있습니다.
    • Monitoring: Micrometer, Spring Boot Actuator를 통해 메트릭 수집 및 모니터링이 용이합니다.
  3. 검증된 안정성 및 성능: Spring 팀이 Netty를 기반으로 수많은 사용 사례를 통해 검증하고 최적화한 솔루션입니다. 직접 구현할 때 발생할 수 있는 잠재적인 버그나 성능 문제를 피할 수 있습니다.
  4. 낮은 유지보수 비용: 코드가 간결하고 표준화되어 있어 새로운 개발자가 합류해도 이해하기 쉽고, 유지보수가 편리합니다.

❌ 단점

  1. 제한된 유연성: Spring Cloud Gateway가 제공하는 방식(Route, Predicate, Filter)에서 벗어나는 매우 특수한 로직을 구현해야 할 경우, 프레임워크의 구조 때문에 제약이 있을 수 있습니다.
  2. 프레임워크 오버헤드: 순수 Netty 코드에 비하면 약간의 프레임워크 오버헤드가 존재합니다. 하지만 대부분의 애플리케이션에서는 무시할 수 있는 수준이며, 생산성 향상으로 얻는 이점이 훨씬 큽니다.

2. 순수 Netty로 직접 구현 시 장점과 단점

✅ 장점

  1. 최고 수준의 성능 튜닝: 메모리 풀링, Zero-Copy 등 Netty의 저수준 기능을 직접 제어하여 성능을 한계까지 끌어올릴 수 있습니다. (단, 이는 최고 수준의 Netty 전문가일 경우에만 해당됩니다.)
  2. 완벽한 제어권 및 유연성: HTTP 프로토콜뿐만 아니라, 게임 서버를 위한 독자적인 TCP 프로토콜, MQTT, gRPC 프록시 등 어떤 종류의 네트워크 애플리케이션이든 원하는 대로 정확하게 구현할 수 있습니다.
  3. 최소한의 의존성: 프레임워크 없이 필요한 라이브러리만 사용하여 매우 가볍게 만들 수 있습니다.

❌ 단점

  1. “바퀴의 재발명”: Spring Cloud Gateway가 제공하는 아래 기능들을 모두 처음부터 직접 만들어야 합니다. 이는 엄청난 시간과 노력이 드는 작업입니다.
    • 설정 파일 기반의 동적 라우팅 엔진
    • 요청 경로, 헤더, 메소드 등을 분석하는 조건부 매칭(Predicate) 로직
    • 필터 체인 및 필터 실행 순서 관리
    • HTTP 요청/응답 객체 파싱 및 조작의 편의성
    • Service Discovery 클라이언트 연동
    • Circuit Breaker, Rate Limiter 구현
    • 보안, 로깅, 메트릭 수집… 등
  2. 극도로 높은 복잡도와 학습 곡선: Netty의 내부 동작(EventLoop, Channel, Pipeline, ByteBuf 등)을 완벽하게 이해해야 하며, 비동기 및 동시성 관련 버그(메모리 누수, 데드락 등)를 다루기가 매우 어렵습니다.
  3. 높은 개발 및 유지보수 비용: 초기 개발 기간이 매우 길고, 코드가 복잡해져서 특정 개발자에게 의존하게 될 가능성이 높습니다. 장기적으로 유지보수가 매우 힘들어집니다.

결론: 언제 무엇을 선택해야 할까?

  • Spring Cloud Gateway를 선택해야 하는 경우 (대부분의 상황):

    • MSA(마이크로서비스 아키텍처)의 API 게이트웨이를 구축하는 경우
    • HTTP/HTTPS 기반의 API를 처리하는 경우
    • 빠른 개발과 안정적인 운영, 쉬운 유지보수가 중요한 경우
    • Spring 생태계를 이미 사용하고 있거나 활용할 계획인 경우

  • 순수 Netty로 직접 구현을 고려해볼 수 있는 경우 (극히 드문 상황):

    • HTTP가 아닌 독자적인 프로토콜을 처리하는 프록시나 게이트웨이가 필요한 경우 (예: 게임, IoT, 금융권의 실시간 통신)
    • 기존 게이트웨이 솔루션으로는 도저히 만족할 수 없는, 나노초(nanosecond) 단위의 극한의 성능 최적화가 비즈니스의 핵심 요구사항인 경우
    • 이러한 시스템을 개발하고 유지보수할 수 있는 세계적인 수준의 Netty 전문가 팀을 보유하고 있는 경우

따라서 일반적인 웹 서비스나 MSA 환경에서 API 게이트웨이를 구축한다면, 고민할 필요 없이 Spring Cloud Gateway를 사용하는 것이 현명한 선택입니다.

스프링 클라우드 게이트웨이 로드벨런서

스프링 클라우드 게이트웨이(Spring Cloud Gateway, SCG) 단독으로 훌륭한 로드밸런서(LB) 역할을 수행할 수 있습니다.

SCG 내부에는 트래픽 분산을 위한 Spring Cloud LoadBalancer (과거 Netflix Ribbon을 대체한 모듈)가 내장되어 있기 때문에, 백엔드 서비스 앞에 별도의 Nginx나 HAProxy, L4 스위치 등을 두지 않아도 직접 여러 노드로 트래픽을 로드밸런싱할 수 있습니다.

유량 제어(Rate Limiting)와 결합하여 SCG를 로드밸런서로 사용할 때의 구성 방법과 작동 방식을 정리해 드립니다.

1. SCG가 로드밸런싱을 수행하는 방법

SCG에서 라우팅 주소(URI)를 설정할 때 http:// 대신 lb:// 접두사를 사용하면, SCG는 이를 단일 주소가 아닌 '로드밸런싱 대상 클러스터’로 인식합니다.

백엔드 노드들의 IP 정보를 SCG가 알아내는 방식에 따라 두 가지로 나뉩니다.

A. 동적 로드밸런싱 (서비스 디스커버리 연동 - 권장)

  • 방식: Kubernetes, Eureka, Consul 등의 서비스 레지스트리와 연동합니다.
  • 설정 예시:
  • 작동: SCG는 my-backend-service라는 이름으로 등록된 실제 노드(IP:Port) 목록을 레지스트리에서 동적으로 가져와서 Round-Robin(기본값) 등의 방식으로 로드밸런싱합니다.

B. 정적 로드밸런싱 (설정 파일에 IP 명시)

  • 서비스 디스커버리 인프라가 없다면, application.yml에 백엔드 노드들의 주소를 직접 하드코딩하여 로드밸런싱할 수도 있습니다.

2. 유량 제어(Rate Limit)와 로드밸런싱의 동작 순서

SCG 내부에서 요청을 처리하는 순서는 다음과 같습니다.

  1. 클라이언트 요청 도착
  2. 유량 제어 필터 (RequestRateLimiter) 실행:
    • 사전에 설정한 기준(예: 사용자 ID별 초당 10건, 클러스터 전체 초당 1000건 등)을 검사합니다. (보통 Redis 활용)
    • 차단 시: 백엔드로 요청을 보내지 않고 즉시 429 Too Many Requests 반환.
  3. 로드밸런서 실행:
    • 유량 제어를 통과한 요청에 한해, Spring Cloud LoadBalancer가 백엔드 노드 3개 중 1개를 선택(예: 10.0.0.2)합니다.
  4. 백엔드로 포워딩: 선택된 노드로 실제 HTTP 요청을 전달합니다.

3. SCG를 단독 LB로 쓸 때 반드시 고려해야 할 설정 (실무 팁)

SCG가 LB 역할을 할 때 장애를 안정적으로 처리하기 위해 반드시 추가해야 하는 필수 필터들이 있습니다.

① 백엔드 장애 대비 Retry (재시도) 필터

SCG가 트래픽을 보낸 노드가 하필 그 순간 다운되었거나 일시적 오류(503 등)를 내뱉을 수 있습니다. 이때 다른 정상 노드로 다시 요청을 보내도록 설정해야 합니다.

② 타임아웃(Timeout) 설정

백엔드 노드가 죽진 않았지만 응답이 무한정 길어질 때, SCG가 이를 빨리 끊어내고 다른 노드나 에러로 처리하게 해야 SCG 자체의 쓰레드가 고갈되지 않습니다.

4. 결론

SCG 단독으로 LB 역할을 수행하는 것은 네트워크 홉(Hop)을 줄이고 아키텍처를 단순화할 수 있는 매우 권장되는 패턴입니다.

  • 앞서 질문하셨던 **“Hazelcast를 이용한 클러스터 전체 유량 제어”**를 1차 관문으로 세우고,
  • 그 관문을 통과한 트래픽을 SCG의 lb:// 스킴을 통해 여러 백엔드 노드로 분산시킨다면,

안정적이고 성능이 뛰어난 마이크로 게이트웨이 아키텍처를 완성하실 수 있습니다. (다만 백엔드 노드의 Health Check를 자동화하기 위해 Kubernetes Service 이름으로 라우팅하거나 Eureka 같은 디스커버리를 함께 쓰는 것이 좋습니다.)

Resilience4j CircuitBreaker

“Resilience4j CircuitBreaker를 함께 쓴다는 것” 은, 한마디로 “특정 백엔드 인스턴스가 느려지거나 고장 났을 때, 게이트웨이 전체가 마비되는 것을 막고 우아하게(Graceful) 장애를 처리하는 안전장치를 단다는 것” 을 의미합니다.

Spring Cloud Gateway(SCG)의 lb://는 트래픽을 분산할 뿐, 트래픽을 받은 인스턴스가 **‘느리게 응답하는 상황(Hang)’**에 대해서는 능동적으로 대처하지 못합니다. 인스턴스가 느려지면 SCG의 쓰레드(커넥션)도 함께 응답을 기다리며 묶이게 되고, 결국 전체 게이트웨이의 성능 저하로 이어집니다(Cascading Failure).

이를 해결하기 위해 Resilience4j를 도입하면 다음과 같은 이점을 얻을 수 있습니다.

1. Resilience4j 적용 시 얻는 3가지 효과

  1. 타임아웃(TimeLimiter) 및 빠른 실패(Fail-Fast): 특정 인스턴스가 설정된 시간(예: 2초) 내에 응답하지 않으면, 더 기다리지 않고 강제로 연결을 끊어버립니다.
  2. 장애 격리(서킷 오픈): 특정 인스턴스(또는 라우트)의 에러율이나 지연율이 임계치(예: 50%)를 넘어가면, '서킷(회로)'을 차단하여 당분간 해당 백엔드로 아예 요청을 보내지 않고 즉각 차단시킵니다.
  3. 폴백(Fallback): 에러 페이지(500, 504)를 클라이언트에게 던지는 대신, 미리 준비된 기본 데이터나 안내 메시지를 반환합니다.

특히 lb:// (로드밸런서)와 Retry 필터, 그리고 CircuitBreaker를 조합하면, “인스턴스 2번이 느려서 서킷브레이커가 타임아웃을 발생시킴 -> Retry 필터가 이를 감지하고 인스턴스 1번으로 재시도함” 과 같은 진정한 인스턴스 단위 장애 격리 및 우회가 가능해집니다.

2. 적용을 위한 코드 및 설정 방법

Spring Boot 3.x / Spring Cloud 2022.x 이상 환경을 기준으로 설명합니다.

Step 1: 의존성 추가 (build.gradle)

Resilience4j를 SCG에서 사용하기 위해 다음 라이브러리를 추가합니다. (SCG는 WebFlux 기반이므로 리액터용 라이브러리가 필요합니다)

Step 2: application.yml 설정

라우트 정보에 CircuitBreaker 필터를 추가하고, 동작 정책(타임아웃, 에러율 등)을 정의합니다.

Step 3: Fallback Controller 구현 (Java)

서킷브레이커가 동작하여 백엔드 호출을 차단하거나 타임아웃이 발생했을 때, 클라이언트에게 반환할 응답을 정의합니다. SCG는 비동기 논블로킹(WebFlux)으로 동작하므로 @RestController를 만들 때 주의해야 합니다.

Step 4: (선택) Java 코드로 직접 설정 세밀화하기

만약 YAML 방식이 아니라 Java Bean으로 서킷브레이커의 기본 동작을 제어하고 싶다면 아래처럼 설정할 수 있습니다.

3. 정리

요약하자면, SCG에서 **lb://**를 통해 부하를 분산하는 상황에서,

  1. Resilience4j의 TimeLimiter가 느린 인스턴스에 대한 연결을 빠르게 끊어 게이트웨이를 보호하고,
  2. 끊어진 요청을 Retry 필터가 받아 정상적인 다른 인스턴스로 넘기며,
  3. 전체 클러스터가 고장 났을 때는 CircuitBreaker가 회로를 차단하고 Fallback 응답을 내려보내는 것.

“단순 부하 분산을 넘어선 인스턴스 단위 장애 격리” 의 완성된 모습입니다.

spring.cloud.discovery.client.simple 설정

다만, 라우트(gateway.routes)를 동적으로 변경하는 것과는 내부 동작 원리에 약간의 차이가 있습니다. 실무에서 이 백엔드 IP 목록(Simple Discovery Client 설정)을 동적으로 갱신하는 두 가지 주요 방법을 설명해 드리겠습니다.

방법 1. Spring Cloud Actuator의 Refresh 기능 사용 (설정 기반)

spring.cloud.discovery.client.simple 설정은 내부적으로 Spring의 @ConfigurationProperties로 바인딩됩니다. 따라서 Spring Cloud Config(설정 서버)를 사용하거나, 실행 환경의 Properties를 변경한 뒤 Actuator의 /refresh 엔드포인트를 호출하면 애플리케이션 재시작 없이 메모리상의 IP 목록이 갱신됩니다.

주의할 점 (로드밸런서 캐시):
IP 목록이 갱신되더라도 즉시 반영되지 않을 수 있습니다. Spring Cloud LoadBalancer(SCG 내부의 LB)는 성능을 위해 디스커버리 결과를 기본 35초 동안 캐싱합니다. 따라서 캐시 만료(TTL) 시간이 지나야 새로운 IP로 트래픽이 넘어갑니다.

방법 2. Custom Discovery Client 구현 (강력 추천 - Hazelcast 연동 시)

이전 질문에서 “공통 데이터는 헤이즐캐스트를 사용하고 DB에서 읽어오는 솔루션” 을 기획 중이라고 하셨습니다. 그렇다면 application.yml의 simple.instances를 사용하는 것보다, 자바 코드로 직접 Discovery Client를 구현하여 DB나 Hazelcast에서 IP 목록을 실시간으로 읽어오도록 만드는 것이 훨씬 유연하고 솔루션에 적합합니다.

YAML의 simple.instances를 완전히 지우고, 아래와 같이 커스텀 로드밸런싱 소스를 만들 수 있습니다.

구현 예시 (Custom ServiceInstanceListSupplier)

특정 서비스 이름(my-backend-service)에 해당하는 IP 목록을 로드밸런서가 요구할 때마다 Hazelcast나 DB에서 실시간으로 제공하는 클래스를 만듭니다.

로드밸런서 설정에 등록하기

위에서 만든 Supplier를 Spring Cloud LoadBalancer가 사용하도록 설정 클래스를 만듭니다.

결론

  1. 단순 테스트나 수동 갱신 목적이라면 Spring의 @ConfigurationProperties 갱신 메커니즘(/refresh) 을 통해 동적 변경이 가능합니다.
  2. 하지만 질문자님께서 구상하시는 독립된 마이크로게이트웨이 솔루션(DB + Hazelcast UI 관리) 을 완성하기 위해서는, YAML 파일에 simple.instances를 적어두는 방식은 한계가 있습니다.
  3. 대신 ServiceInstanceListSupplier 또는 ReactiveDiscoveryClient 인터페이스를 직접 구현하여, 관리 UI에서 IP를 추가/삭제하면 Hazelcast를 통해 즉각적으로 LB의 IP 목록이 갱신되도록(Zero Downtime) 구현하시는 것이 가장 완벽한 설계입니다.

스프링으로 동적 등록 구현하기

스프링 컨트롤러와 서비스를 활용하면 관리 UI나 외부 시스템(CI/CD, 모니터링 툴)에서 API 호출 한 번으로 실시간 노드 추가/삭제가 가능한 완벽한 동적 환경을 구축할 수 있습니다.

질문하신 시나리오(컨트롤러 + 서비스 + 헤이즐캐스트)를 바탕으로 한 구현 설계를 단계별로 가이드해 드립니다.

1. 전체 흐름도

  1. 관리자(UI): POST /api/admin/instances 호출 (새로운 IP 등록 요청)
  2. Controller: 요청을 받고 DiscoveryService 호출
  3. DiscoveryService: Hazelcast(IMap) 에 인스턴스 정보 저장 (클러스터 전체 공유)
  4. LoadBalancer: 트래픽 발생 시 Hazelcast에서 최신 IP 목록을 읽어와 LB 수행

2. 단계별 구현 코드

① 데이터 구조 정의 (DTO)

먼저 등록할 인스턴스 정보를 담을 간단한 클래스를 만듭니다. (Hazelcast 저장을 위해 Serializable이 필요합니다.)

② 인스턴스 관리 서비스 (DiscoveryService)

Hazelcast를 저장소로 사용하여 인스턴스 목록을 관리하는 서비스입니다.

③ 관리용 API 컨트롤러

UI에서 호출할 엔드포인트입니다.

④ 로드밸런서와 연동 (Custom Supplier)

이제 SCG가 트래픽을 보낼 때 위 서비스(Hazelcast)에서 IP 목록을 가져오도록 연결합니다.

⑤ 로드밸런서 설정 등록

모든 서비스(lb://*)에 대해 위 메커니즘을 적용합니다.

3. 이 방식의 솔루션적 가치

  1. 완전한 동적 운영: 관리 UI에서 버튼 클릭 한 번으로 새 서버를 투입하거나 장애 노드를 즉시 트래픽에서 제외할 수 있습니다. (Zero-Downtime)
  2. 상태 공유의 무결성: Hazelcast를 사용하기 때문에 게이트웨이 노드가 100대여도 API 호출 한 번에 모든 게이트웨이의 로드밸런싱 목록이 동시에 바뀝니다.
  3. 확장성: 추후에 "헬스체크 대행 서비스"를 별도로 만들어, 특정 노드가 응답이 없으면 자동으로 이 컨트롤러나 서비스를 호출하여 노드를 제거(Deregister)하는 자동화 로직을 붙이기 매우 쉽습니다.
  4. 관측 가능성: 컨트롤러에 GET /instances API만 추가하면 현재 게이트웨이가 바라보고 있는 백엔드 노드 현황을 UI에 실시간으로 시각화할 수 있습니다.

결론적으로, YAML 설정을 사용하는 것보다 컨트롤러+서비스+Hazelcast 조합이 질문자님이 구상하시는 ‘관리형 마이크로게이트웨이 솔루션’ 에 훨씬 더 부합하는 전문적인 설계입니다.

Annotation processor

Annotation

어노테이션은 메타데이터의 한 종류이다. @Override, @Component와 같이 아주 익숙한 표현이 있는데 이처럼 @가 붙어있는 단어를 어노테이션이라 한다. 클래스, 메소드, 변수 등에 어노테이션을 붙일 수 있다. 어노테이션은 메타데이터라고 말할 것과 같이 소스코드 상에서 주석처럼 읽을 수도 있고, 컴파일러가 어노테이션을 찾아 컴파일 단계에서 특정 동작이 이루어지도록 할 수있다. 런타임에 JVM에 의해 특정 동작에 사용 될 수도 있다.

Custom Annotation

스프링 부트에서 익숙하게 사용하는 어노테이션들을 이용하면 예상대로 그대로 동작한다. 여기서 사용하는 어노테이션은 스프링 부트를 만드는 사람들이 만든 어노테이션이다. 이처럼 실제 동작하는 어노테이션을 직접 만들어보자.

위 처럼 @interface 로 타입을 정의 하고 id와 menu를 정의한다. 이것들은 변수 처럼 사용된다. defualt는 아무런 값을 지정해 주지 않았을 때 들어가는 값이다. @target 은 어노테이션을 어디에 적용할 지에 대한 내용이다. 위 코드는 메소트에 적용할 어노테이션이라는 것을 나타낸다. @Retetion 은 어노테이션을 언제 적용할지를 정해주는 값이다. RUNTIME은 코드 실행 중에 적용하겠다는 의미다.

실제 어노테이션을 사용해 프로그램 실행 중에 메소드의 정보를 가져와서 메소드이름과 메소드에 적용되어 있는 어노테이션 이름을 출력해 보았다. 다음은 어노테이션을 적용해 컴파일시 코드를 작성하는 어노테이션을 만들어 본다.

Annotation Processor

어노테이션 프로세서는 컴파일 단계에서 어노테이션을 검사하여 코드베이스를 검사하거나 수정, 생성하는 일을 한다. 코드를 생성하거나 수정하는 일을 이용해 개발자는 반복적인 소스코드 작업을 줄일 수 있다. 실제 코드를 생성하는 어노테이션을 만들어보자.

@Tartget에 TYPE을 주어 class에 어노테이션을 지정하겠다고 명시한다. @Retention은 SOURCE를 준다. 아래 코드는 어노테이션이 적용된 클래스들을 찾아서 이 클래스와 관련된 다른 소스코드를 생성해주는 annotation procees를 실행하는 코드이다.

@AutoService를 추가해 어노테이션 프로세서를 지정해준다. init 메소드는 처음 한번 실행된다. process가 실제로 annotation processor가 동작할 때 일하는 메소드이다. 위 코드는 실제 업무에 쓰이는 코드를 임의로 수정했기 때문에 동작을 보장 할 수 없지만 대략 이런식으로 소스가 구성된다 정도만 이해하자. 이렇게 작성한 코드를 아래와 같이 pom.xml에 설정을 추가한 후 jar 로 컴파일 한다. gradle로 설정하는 법은 따로 공부해보자.

빌드한 결과를 실제 적용할 프로젝트로 가져와서 의존성을 위처럼 추가한다. 클래스에 어노테이션을 적용하고 빌드하면 빌드 결과물에 소스가 생성되고 컴파일된 클래스가 추가되어있는 것을 확인할 수 있다.

  • 자바 어플리케이션에서 관계형 데이터베이스를 사용하는 방식을 정의한 인터페이스
  • ORM을 위한 자바 EE 표준

ORM(Object-Relation Mapping)

  • 객체-관계 매핑
  • 객체를 통해 간접적으로 DB에 접근
  • 객체와 DB 데이터를 자동으로 매핑
  • SQL을 쓰지 않고 메서드로 데이터 조작
  • Persistant API
  • JPA, Hibernate

Persistance

  • 데이터를 생성한 프로그램이 종료되어도 사라지지 않는 데이터의 특성
  • 메모리 상에 존재하는 데이터는 프로그램이 종료될 경우 사라지기 때문에 데이터를 파일이나 DB에 저장하여 영속성을 부여

Persistance Layer

  • 프로그램 아키텍처에서 데이터에 영속성을 부여해주는 계층
  • JDBC를 이용해 구현이 가능, Persistance Framework를 사용
  • Persistance Framework는 SQL Mapper와 ORM이 있음

Spring Data JPA

  • 스프링에서 JPA를 편리하게 사용할 수 있도록 지원하는 프로젝트
  • CRUD 처리를 위한 공통 인터페이스
  • 인터페이스만 작성하면 동적으로 구현체를 생성해서 주입해줌

JpaRepository

  • Spring Data JPA에서 CRUD 처리를 위한 공통 인터페이스
  • MongoRepository는 mongodb를 사용하기 위한 JpaRepository

Java String, StringBuffer, StringBuilder

Java에서 문자열을 이용해 이것 저것을 할 경우가 있다. 성능을 내려면 String가지고는 부족하다. 그렇다… 부족하다.

String

  • String은 immutable(변경불가)이다.
  • String에 문자열을 더하거나 편집 할 경우 그 결과가 새로운 메모리 공간에 할당된다.
  • 기존 문자열은 가비지 컬렉터의 대상이다. GG
  • 조회는 빠르다.

StringBuffer

  • StringBuffer는 mutable(변경가능)이다.
  • 문자열을 편집할 경우 메모리 영역을 늘려 문자열을 저장한다.
  • thread safe 하다.

StringBuilder

  • StringBuilder는 StringBuffer와 유사하다.
  • 단, thread safe 하지 않다.
  • StringBuffer 보다 연산이 좀 더 빠르다.

Java Sort Algorithm

자바를 이용해 정렬 알고리즘에 대해 알아보자.

Bubble Sort

  • 버블 정렬
  • 서로 인접한 두 원소를 검사하여 정렬한다.
  • 아래 코드는 오름차순 정렬이다.

Selection Sort

  • 선택 정렬
  • 버블 정렬과 마찬가지로 입력 된 배열이외의 저장공간을 필요로 하지 않는다.
  • 버블 정렬보다는 빠르다.
  • 한번 배열을 돌때마다 가장 큰(혹은 작은) 값의 인덱스를 가지고 위치를 선정하여 바꾼다.

Insertion Sort

  • 삽입 정렬
  • 손안의 카드를 정렬하는 것과 유사한 방법
  • 두번째 요소부터 비교를 시작한다. 하나의 요소의 앞부분(혹은 뒷부분)에 있는 요소들과 비교하여 적절한 위치에 넣는다.
  • 버블 정렬, 선택 정렬 보다는 빠르다

Shell Sort

  • 셸 정렬
  • 삽입 정렬을 보완한 알고리즘으로 Donald L. Shell 이라는 사람이 제안
  • 어느 정도 정렬 된 배열에 대해서 삽입 정렬이 대단히 빠른 것에 착안
  • 삽입 정렬에 비해 빠르고, 구현도 비교적 간단하다.
  • 삽입 정렬과는 달리 셸 정렬은 전체 리스트를 한번에 정렬하지 않는다.
  • 정렬해야 할 리스트의 k번째 요소를 추출해서 부분 리스트를 만든다. k는 gap(간격)이라 부른다.
  • 간격의 초기값: (정렬 할 요소의 수)/2
  • loop가 돌때 마다 k를 절반으로 줄인다.
  • 간격을 절반으로 줄일 때 짝수가 나오면 +1을 해서 홀수로 만든다.
  • 간격이 1이 될 때 까지 반복한다.

Merge Sort

  • 병합 정렬
  • John von Neumann(존 폰 노이만)이라는 사람이 제안한 방법
  • 이 정렬은 안정 정렬에 속하며, 분할 정복 알고리즘의 하나이다.
  • 레코드를 array로 구성하면 임시배열이 필요
  • 레코드를 Linked List(연결 리스트)로 구성하면 더 좋다.

Quick Sort

  • Charles Antony Richard Hoare(찰스 앤터니 리처드 호어)가 개발
  • 불안정 정렬에 속함 비교 정렬에 속함
  • 합병 정렬과 같이 분할 정복 알고리즘이나 리스트를 비균등하게 분할함
  • 리스트의 한 요소를 선택, 이를 피벗(pivot)이라고 부름
  • 피벗을 기준으로 피벗보다 작은 요소들은 모두 피벗의 왼쪽으로 옮겨지고, 피벗보다 크면 오른쪽으로 옮겨진다.
  • 피벗을 제외한 왼쪽과 오른쪽 리스트를 다시 정렬한다.
  • 이 때 왼쪽과 오른쪽 리스트에서 피벗을 선택하여 동일한 과정을 거친다.
  • 리스트의 크기가 0 혹은 1이 될때까지 반복한다.
  • 정렬된 리스트일 경우 퀵정렬의 불균형 분할이 수행시간을 늘릴 수 있다.
  • 자바의 Arrays.sort()는 위의 단점을 피하기위해 수정 된 퀵 정렬을 사용한다.

Heap Sort

  • 힙 정렬
  • 최대 힙 트리 혹은 최소 힙트리를 구성해 정렬하는 방법
  • 내림차순 정렬은 최대 힙, 오름차순 정렬은 최소 힙을 구성한다.
  • 최대 힙 트리를 만들고 요소를 하나씩 배열

참고

  • 안정(stable) 정렬과 불안정(unstable) 정렬
  • 정렬되지 않은 상태에서 같은 키값을 가진 원소의 순서가 정렬 후에도 유지되면 안정 정렬
  • 7 다이아, 4 하트, 2 클로버, 4 스페이드 를 숫자를 이용해 정렬 할 때 4 하트와 4 스페이드의 순서가 바뀌거나 바뀌지 않음으로 안정과 불안정을 구분할 수 있다.
  • 자료구조 힙(hepa)
  • 완전 이진 트리의 일종으로 우선순위 큐를 위하여 만들어진 자료구조
  • 최대값 또는 최소값을 빠를게 찾을 수 있도록 만들어진 구조
  • 힘은 일종의 반정렬 상태를 유지 - 큰 값이 상위 레벨에 있다.
  • 힙트리는 중복된 값을 허용
  • 첫번째 index는 1, 0은 사용하지 않음
  • 특정 위치의 노드 번호는 고정
  • 왼쪽 자식의 index = (부모 index)*2
  • 오른쪽 자식의 index = (부모 index)*2 + 1
  • 부모 index = (자식 index)/2
