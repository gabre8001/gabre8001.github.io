Bash 배경

  • Shell 이란

    • application을 실행시키는 도구
    • shell 에서 내리는 명령어는 해석된 뒤에 자시 프로세스로 만들어져 실행됨
    • Bourne shell /bin/sh 표준 셸 POSIX
      • Stephen R. Bourne이 작성 1977
      • 기본 유닉스 셸
      • 기능이 미약. 대부분의 기능이 외부 명령어로 대체
    • C shell /bin/csh
      • Berkly의 Bill Joy가 작성. BSD UNIX에 포팅
      • C 언어와 비슷하 문법으로 C shell 이라 명명
      • sh와 너무 달라 외면. 후에 개량형으로 TC sh 등장
    • Korn shell /bin/ksh
      • Bell Lab의 David Korn이 작성 1983
      • csh 장점과 bourne shell 기능을 모두 포함
      • vi prompt mode 지원
      • 다양한 변수형, 사칙 연산 기능 확장
      • 현대적인 상용 유닉스 기본셸
    • bash /bin/bash 가장 유명하 셸
      • Bourne again shell. born again sh
      • GNU에서 발전시킨 개량판 1989
      • 현대적인 셸의 기능 대부분이 추가되어 Korn shell과 호환
      • Korn shell을 벤치마킹
      • build in cmd로 대체 = 성능 향상

      • Paul Falstad 작성 1990

bash shell 작동원리

  • 셸에서 실행되는 UNIX 명령어는 sub-shell 이 생성되어 실행

  • 서브셸이란 fork-exec로 만들어지는 child process 셸

  • 부모 프로세스는 자식 프로세스의 exit code 를 읽어햐한다.

  • 자식이 성공했는지 실패했는지 알수 있다. 0이면 성공 아니면 실패

  • zombie

    • 부모 프로세스가 종료된 자식 프로세스의 exit status르 처리하기 전의 상태
    • 자식 프로세스를 zombie process 혹은 defunct process 라고 부름
    • 모든 프로세스의 종료는 zombie 상태를 거쳐서 해소되기 때문에 프로세스가 종료되는 정상적인 과정중에 하나.
    • 좀비 프로세스가 생긴다는 건 부모 프로세스에 문제가 있다는 것!

  • echo $? 이전 명령어의 리턴값

  • rm --force 옵션에 대하 오래
    • UNIX command 특히 rm과 같으 optional하 기능에 --force 옵션이 존재하는 것은 child process로 작동할 때 exit code르 조작하기 위함.

  • make 외에 gradle이나 다른 빌드 툴도 동일한 기능으로 작동하느 것들이 많다.

  • 프로세스의 main 함수의 리턴값 == exit code

  • 부모 프로세스가 자식 프로세스가 어떻게 종료되었는지 알수 있다.

  • shell script도 정상 작동이 아닌 경우 return 값을 0이 아닌 값을 줘야한다.

  • 셸의 명령 기능을 조합하여 batch 처리를 목표로 한다.

    • 외부 명령어와 built-in 명령어르 조합
    • 그 외에 perl(정규표현식 사용 많을 때), python(c언어와의 연동을 생각하면), ruby와 같으 스크립트 언어 사용

Linux Network System

  • 네트워크 기본 용어
    • hostname: primary hostname, FQDN
      • abc.redhat.com 에서 given name + family name에 비유하면 redhat.com은 family name으로 domain address라 불리기도한다.
      • domain을 제외한 호스트 이름 혹은 domain을 포함한 FQDN
      • FQDN(Fully Qualified Domain Name) 도메인 내에서 유일하게 구별 가능한 이름. DNS server에서 정함
      • 도메인 주소는 체계적인 구조를 가짐 devel.fclinux.or.kr
      • localhost는 항상 자기 자신을 의미하는 주소와 매핑되는 특별한 hostname. (IPv4= 127.0.0.1, IPv6= ::1)
    • TCP/IP: IP address(ipv4, ipv6), subnet mask, gateway
      • IPv4 32bit 주소 체계. 8bit씩 끊어서 xxx.ooo.xxx.ooo
      • IPv6 128bit 주소 체계. 무지 많다.
      • IPv4-mapped IPv6 ::ffff:IPv4_addr
      • dual stack IPv4와 IPv6를 혼용해서 사용하기 위한 설계. 대부분 OS 지원 (win xp 제외)
      • A,B,C 클래스 뭐시기는 없어짐
      • CIDR(Classless Inter-Domain Routing) IP클래스와 상관럾이 서브넷을 지정하여 자르는 것을 의미. 서브넷 마스크 비트 크기.
      • public IP 공인주소(인터넷에서 유일한 주소), private IP(인터넷에 직접 연결되지 않은 주소)
    • NIC: Network Interface Card == 랜카드
    • Wired Network (Wired connection): 유선 네트워크(유선 연결)
    • Wireless Network (Wireless connection): 무선 네트워크(무선 연결)
    • LAN: Local Area Network
    • WAN: Wide Area Network
    • SELinux(Security Enhanced Linux) 커널 레벨에서의 중앙 집중식 보안 기능
    • SELinux 보안 레벨
      • enforcing(강제): 보안설정에 걸리면 강제로 막음(많은 보안 지식필요)
      • permissive(허가): 보안설정에 걸리면 허용하되 로그 출력(실습시 사용 권장)
      • disable(비활성): SELinux 사용하지 않음

Linux NetworkManager

  • NetworkManager는 deamon으로 작동하면서 network configureation을 수행하고, 자동으로 network connection을 관리(연결 감지, 해제, 재시도)를 수행한다. Dbus 기반으로 동적 상태를 감지할 수 있기 때문에 다른 애플리케이션이나 deamon들에게 네트워크 정보를 제공하거나, 관리를 위한 권한을 줄 수도 있다. 또한 명령어를 사용하여 systemd 기반의 다른 Linux distribution들에게도 동일한 방식의 경험을 제공할 수 있다. Ethernet, Wi-Fi, Mobile broadband 등 다양한 기능들에게 플랫폼을 제공하므로, 네트워크 연결 관리가 좀 더 쉬워진다.

  • networks device naming
    • eth#[:n] eth0, eth1, eth0:0 등 으로 나타남. 옛날 표현 방식
    • Consistent Network Device Naming(2009)
      • prefix en(ethernet), wl(wireless lan), ww(wireless wan)
      • o<index> on-board device index number
      • s<slot>[f<function>][d<dev_id] hotplug slot index number
      • x<mac> MAC address
      • p<bus>s<slot>[f<function>][d<dev_id] PCI geographical location
      • p<bus>s<slot>[f<function>][d<dev_id][..][c<config>][i<interface>] USB port number chain
      • enp5s0 -> ethernet & pci number 0

File System

  • FS는 OS의 매우 큰 부분을 차지한다. read, wirte 작업을 수행

  • FS는 넓게 보면 데이터 베이스의 일종. 파일을 저장하는 데이터 베이스

  • 종류

    • ext4: 리눅스 고유 파일 시스템
    • ext3, ext2: ext4의 이전 버전
    • xfs: 실리콘 그래픽스의 저널링 파일 시스템(RHEL7부터 기본 파일시스템)
    • zfs: 솔라리스의 기본 파일 시스템, CoW지원
    • Btrfs: zfs와 비슷한 컨셉으로 개발된
    • ntfs: 윈도우즈 NT 계열에서 사용하는 파일 시스템
    • fat32: FAT 32비트 버전
    • exfat: FAT 32비트 확장 버전

  • 새로운 FS 만드는 법

    • partitioning: fdisk, parted
    • file system: mkfs, fsck or xfs_*
    • mount: mount/unmount. /etc/fstab

  • Partition

    • physical/logical 구획이 있다. 현대 OS는 logical partition을 선호
    • logical portition을 사용하면 물리적 한계를 가진 디스크를 묶어서 보다 큰 용량의 파티션을 만들 수 있다. logical volume(LVM)
    • physical disklabel type
      • DOS 방식: 2TB 제한, fdisk 사용
      • GPT 방식: 무제한 용량, parted/gparted for gdisk 사용
    • Dos partition 종류
      • primary partition(주 파티션) 하나의 디스크에 4개 까지 만들 수 있음
      • extended partition(확장 파티션) 4개 이상 파티션이 필요할 때 primary partition 대신 1개 만들 수 있음. extended partition은 여러 개의 logical drive로 나눌 수 있음.

  • fdisk

    • fdisk -l 현재 FS 리스트를 출력. 요즘 lsblk 사용
    • block device
      • 저장 장치를 이르는 용어
      • /dev/sd SCSI disk, serial type SATA, USB
      • /dev/hd IDE disk
    • fdisk /dev/sda
      • a: 부트 활성 플래그를 지정
      • d: 파티션 삭제
      • l: 알려진 파티션 ID
      • n: 새로운 파티션을 생성
      • p: 현재 파티션 상태를 출력
      • t: 파티션 ID(type)을 변경
      • q: 변경된 상태를 저장하지 않고 종료
      • w: 변경된 상태를 저장하고 종료

  • Make File System

    • 저장 장치는 아주 커다란 종이로 비유하면 파티셔닝은 종이를 몇개로 자르는 작업이다.
    • 파티션 된 종이를 A4와 같은 규격화된 형태로 잘라서 분철하는 것과 유사.
    • 분철을 하면 권수와 페이지 번호가 만들어짐. 인덱싱 가능
    • 분철 방식, 밑줄, 칸의 간격 자물쇄 설치 등에 대한 세부 방식은 file system type에 따름 e.g. ext4, xfs ..
    • ext4
      • 저널링 지원
      • 1EB volume 지원
      • 연속된 파일의 접근 및 작은 파일들의 접근이 빠른 장점
      • 저성능 I/O에서 효율이 높다(IoT 기기)
      • 삭제된 파일 복구 가능
    • XFS
      • 저널링 기반 대용량 파일 시스템
      • SGI 실리콘 그래픽스사에서 개발 IRIX 유닉스에서 사용. 후 오픈소스로 이전
      • 2000년 초반에 포팅되어 많은 기간동안 테스트됨. 대부분의 고성능 서버형 리눅스 배포판의 기본 파일 시스템으로 사용됨. Debian, RHEL7
      • Online 상태(mounted)에서 확장이 가능
      • 대용량 파일 처리시 성능이 좋음
      • kernel service(xfs*)가 관리
      • 삭제된 파일을 복구 할 수 없다.
      • 깨진 파일 시스템 복구를 위한 명령어로 xfs_repair를 제공
    • mkfs
      • make file system 명령어
      • mkfs [-t fstype] [option] <device>
      • lsblk 추가된 disk 확인 인식이 안되있으면
      • ls /sys/class/scsi_host/ | while read hostdev ; do echo “- - -“ /sys/class/scsi_host/$hostdev/scan ; done 입력 후
      • lsblk 로 블락디바이스 확인후 fdisk /dev/sdb 로 새로운 파티션 생성
      • mkfs -t ext4 /dev/sdb2 혹은 mkfs.ext4 /dev/sdb2 파일 시스템 생성
    • fsck
      • fsck [-ARNV] [options] [-t fstypr] <device>
      • file-system checking 일반적으로 offline 된 FS 체크
      • fsck -y -t ext4 /dev/sdb2 혹은 fsck.ext4 -y /dev/sdb2 로 파일 시스템 체크
      • echo $? 이전 명령어(fsck)의 main() 함수의 리턴값(RC: return code)을 확인하여 검사 결과 확인
    • 스왑 공간
      • swap in/out을 위해 RAM의 최대 2배까지 필요
      • mkswap(스왑 공간 작성:포맷)
        • mkswap -L swapfs2 /dev/sdb1 스왑 파티션에 공간 작성
        • dd if=/dev/zero of=./swapfile1 bs=1024 count=262144 입력후 mkswap ./swapfile1 일반 파일을 스왑 공간으로 사용
      • swapon: 스왑 활성 사용
        • swapon /dev/sdb1, swapon ./swapfile1, swapon -L swapfs2
        • cat /proc/swaps 스왑 공간 확인
        • grep swap /etc/fstab 등록된 스왑 공간 확인
      • swapoff: 스왑 비활성
      • 레이블이 한글로 설정될 경우 영문으로 다시 바꾸어 줘야함

  • mount

      • noatime atime 사용 안함

      • :%s,/home/linux,/home/unix,g

      • :%s/CR입력//g 혹은 :%s/\r//g

      • <CTRL-V> o###: ###에 8진수를 사용하여 ASCII 입력

File 관련 명령어

  • pwd - print working directory

  • UNIX file mode

    • file mode bit: 유닉스 파일 권한을 나타내는 3 + 9 bit 체계
    • octal mode: 8진수로 표기되는 UNIX file mode
    • rwxrwxzxrwx: 3칸씩 각각의 owner, group, others 로 나눈다
    • rwxr-x— == 750
    • r == 4 readable
    • w == 2 writable
    • x == 1 executable
    • directory 인경우 400 인 경우 파일 내용 볼 수 있음
    • 파일생성시 기본값은 umask 022 를 뺀 755 가 됨
    • 권한 바꾸기 - chmod 664 testdir / rw-rw-r– r은 목록에서 파일을 볼수 있고, x는 파일에 access할 수 있음.
    • chmod -x hello / chmod +x hello / chmod u+x g=rw ..

    • magic data: /usr/xhare/file/magic

    • ctime: change time - file의 meta-data가 변경된 시간

  • stdio

    • standard input output
    • file channel: fild에 입출력하기 위한 통로, 표준화된 입출력 방식을 통하도록 하는 가상화 레이어

    • 0: stdin - standard input 표준입력
    • 1: stdout - standard output 표준출력
    • 2: stderr - standard error 표준에러

link

  • ln - make links

  • inode - 파일의 메타 정보 및 관리용 객체

    • 파일은 고유의 inode를 1개 가지고 있다.
    • disk partition(or volume)내에서 유일한 식별자. 다른 볼륨이나 파티션에서는 의미 없음
    • file meta data - 시간 관련 정보, 사이즈, 소유권, 권한 등등

  • 하드 링크 - hard link - inode를 가리키는 링크

    • 같은 inode 를 가리키므로 동일 파티션 내에서만 생성 가능
    • regular file 만 가능 directory, device file 등은 hard link를 생성 할 수 없음. 실체를 가진 파일
    • ln

  • 심볼릭 링크 - symbolic link - 특정 파일의 위치를 가리키는 기능을 하는 링크(바로가기)

    • symlink 위치만 가리키므로 다른 파티션도 가능, 모든 종류의 file에 만들 수 있다.
    • 가리키는 대상의 UNIX file mode를 따라가므로 symlink 권한은 777이다. 크게 의미 없음
    • ln -s

  • readlink

    • symlink가 여러 단계로 가리키는 파일이 있을 수 있다. 이 canonical path를 따라가는 기능
    • readlink -f - 마지막 링크를 제외한 모든 링크가 존재할 떼 성공
    • readlink -e - 모든 링크가 존재할 때 성공

  • 사용 예

    • 매일 추천 음악 100에 해당하는 파일을 제공할 때, 매일 파일을 복사하지 않고 symlink가 가리키는 canonical path를 변경하며 사용한다.
    • 라이브러리 버전 관리도 유사하게 사용. /usr/lib 내부 확인해보자.

  • canonicalization

    • canonical이란 컴퓨팅 환경에서 실체를 가지는 standard, official의 의미를 가진다.
    • 예를 들어 상대경로를 절대 경로로 바꾸어 주는 과정, localhost 주소를 127.0.0.1 로 바꾸는 것 등을 canonicalization 한다고 표현한다.
    • symlink는 실체를 가리키는게 아니기 때문에 끝까지 따라가야 canonical path를 알수 있다.

process

  • ps - process status

    • pid: process ID
    • TTY: terminal ID
    • TIME: CPU 시간(누적 시간) = CPU를 점유했던 시간의 누적값. 현실 시간이 아님
    • CMD: command = 프로세스 이름. argv[0]를 의미

  • ps -a: ??

  • ps -f: full-format

    • UID: 해당 프로세스의 소유권자 ID. User ID
    • PPID: 부모 프로세스 ID (parent pid)

    • F: 프로세스 플래그

    • kill- 프로세스에 시그널을 send 하는 기능. 시그널에 따라 프로세스를 제어.
    • 시그널 종류는 kill -l 통해 확인.
      • SIGHUP: Hang Up
      • SIGINT: Interrupt
      • SIGQUIT: Quit <CTRL->. core 발생
      • SIGKILL: kill 강제로 죽임
      • SIGSEGV: Segment violation. 메모리 침범으로 이상 종료
      • SIGTERM: Terminate 종료 요청
      • SIGTSTP: Temporary Stop
    • kill 13011 - PID 13011 프로세스에 SIGTERM 시그널 보냄
    • kill -QUIT 12012 - SIGQUIT 시그널 보냄
    • kill -9 12013 - 9번 시그널(SIGKILL) 보냄 최후의 수단! SIGTERM을 기본적으로 시도

  • job control

    • fore-ground processs - 현재 session에서 제어 터미널(controlling terminal)을 가진 프로세스
    • back-ground process - 현재 session에서 제어 터미널(controlling terminal)을 잃어버린 프로세스
    • CTRL-z
      • SIGTSTP( signal - temporary stop) 시그널을 fore-ground 프로세스에 전달
      • 프로세스를 잠시 정지시킴 = 결과적으로 back-ground에 stpped 상태로 존재
    • session - 세션은 멀티유저 시스템에서 통신 객체(seat of remote)를 구별하기 위함.
      • 세션은 제어 터미널(Controlling terminal)을 가질 수 있다.
      • SID == PID 가 같은 프로세스를 Session Leader 라 부른다. 세션 리더로부터 파생된 자식 프로세스는 모두 같은 세션을 가진다.
      • 세션에 속한 controlling terminal을 소유할 수 있다는 의미 == fore-ground process
      • 로그아웃시 세션이 파괴되면서 세션에 속한 프로세스들은 모두 종료
    • controlling terminal
      • 사용자의 제어를 받는 터미널 장치
      • CUI에서 멀티 태스킹을 위한 제어 방법
      • 제어 터미널을 소유한 프로세스는 키보드 입력을 가진다. 이런 프로세스를 fore-ground process라 부른다.
      • 하나의 session에서 fore-ground process는 최대 1개
      • 제어 터미널 규격 TTY 필드 부분을 보면 두가지 방식
        • pts/#: UNIX98 Pseudo terminal system - SUSv2
        • tty#: console terminal
      • session에서 제어 터미널을 가지지 않는 경우 ps의 TTY 필드에 ? 로 나타난다.
    • process group
      • process group leader: processgroup ID == PID
      • 프로세스 그룹 리더는 시그널 전파 기능을 가짐.
      • kill 에서 음수의 PID 번호는 프로세스 그룹에 시그널을 전파하는 것.
    • deamon 이란??
      • Orphan process( and Session leader)
      • stdio를 모두 /dev/null로 리다이렉션을 걸고
      • control terminal 을 가지지 않는 프로세스. TTY 값 ?이다.
    • interrupt key
      • : SIGINT(Interrupt Signal) 시그널을 fore-ground 프로세스에 전달. 프로세스 그룹 리더가 받을 경우 그룹에 속한 모든 프로세스에 시그널 전파
      • <CTRL->: SIGQUIT(Quit Signal) 시그널을 fore-ground 프로세스에 전달. 동작 방식은 위와 같으나 core 발생
    • commands

      • bg %#: 정지된 백그라운드 프로세스를 back-ground에서 running 상태로 변경

linux booting sequence

  • 경고! init을 이용한 부팅은 지금은 거의 쓰이지 않는다.
  • 그냥 이런거 있구나 정도로 알아두자.

전원 스위치 ON !

  • 시스템 전원 공급하면 메인보드의 ROM-BIOS에 있는 BIOS 프로그램 자동실행
  • BIOS 프로그램은 전원 공급과 함께 메모리의 특정 번지(예:FFFF0H)에 자동 로드 됨
  • CPU는 전원공급과 함께 특정 번지(예:FFFF0H)의 BIOS 프로그램을 자동 실행

BIOS 프로그램들의 실행 내용들

  • 자체 진단 기능(POST): CMOS 검사, CPU, MEMORY, 그래픽 카드, 키보드, 마우스 등 각종 장치들의 이상유무를 검사하고 이 장치들을 초기화 시킴
  • POST 과정이 이상없이 진행완료되면 검색된 부팅매체에서 부트로더(예: GRUB, LILO)를 불러들임
  • 예를 들어 하드디스크가 부팅매체로 선택되었다면 하드디스크의 부팅파티션에 있는 0번섹터 (대부분 MBR이라고도 함)에 있는 부트로더(Boot Loader GRUB)을 읽어들임
  • 부트로더(GRUB)가 메모리에 적재되면 BIOS는 종료되고, 시스템 제어권은 부트로더가 가짐

부트로더(GRUB)의 실행

  • GRUB은 실행과 함께 /boot/grub/grub.conf 파일을 읽어서 어떤 부팅메뉴(커널)로 부팅을 할 것인가를 결정(자동 또는 사용자 선택)
  • GRUB은 커널(kernel) 이미지를 불러들임
  • 시스템 제어권을 커널에게 넘겨줌

커널의 로딩

  • 커널은 swapper 프로세스(PID 0번)를 호출
  • swapper는 커널이 사용할 각 장치드라이브들을 초기화하고 init 프로세스 (PID 1번)를 실행
  • init 프로세스가 실행되면서 /etc/inittab 파일을 읽어들여서 그 내용들을 차례대로 실행

init 프로세스의 실행

  • 이후 과정은 init 프로세스의 실행 내용(/etc/inittab 파일의 실행 내용들)
  • 즉, 로그인 프롬프트가 나오기 까지 init 프로세스에 의해서 실행되는 내용
  • id:3:initdefault: ~~ 시스템 부팅 레벨을 결정하는 행
  • si::sysinit:/etc/rc.d/rc.sysinit ~~ /etc/rc.d/rc.sysinit 파일 실행, 시스템 초기화하는 스크립트들이 있음
  • l3:3:wait:/etc/rc.d/rc 3 ~~ 부팅레벨에 따라 rc3.d 와 같은 파일들이 실행
  • ca::ctrlaltdel:/sbin/shutdown -t3 -r now ~~ CTRL + ALT + DEL 키를 동시에 눌렀을 때 실행될 내용을 설정한 행(일종의 magic key)
  • 1:2345:respawn:/sbin/mingetty tty1 ~~ tty 1부터 6까지 행이 있음, 부팅의 마지막 단계로 6개의 가상 콘솔이 실행 ALT+F1~F6 누르면 해당 번호의 가상콘솔 보여짐
