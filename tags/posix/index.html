File 관련 명령어

  • pwd - print working directory

  • cd - change directory e.g. cd ~/work 경로 생략시 홈 디렉토리로

  • 절대 경로 - /(root directory) 로 시작하는 경로 e.g. /usr/bin

  • 상대 경로 - .(현재 디렉터리)로 시작하는 경로 e.g. ./../../tmp ../../tmp

  • ls - list file

  • mkdir - make directory e.g. mkdir -p work/testdir

  • rmdir - remove directory 이 명령의 경우 디렉토리가 비어있어야하기 때문에 rm -rf 사용

  • cp - copy e.g. cp ~/.bashrc ~/.bashrc_ex

  • mv - move, rename e.g. cp ~/.bashrc_ex ~/.old_bashrc

  • rm - remove e.g. cp ~/.old_bashrc

  • chown, chgrp - change owner/group - root 유저만 가능

  • chattr - change attribute 별로 안중요

  • stat [option] - file meta data 출력

  • touch - 파일의 메타 정보 업데이트. flie이 없을 경우 빈 파일 생성

  • find - 파일 찾기 및 특정 동작 수행

    • find . -mtime -1 -size +1M
    • find . -name ‘*k.dat’ -size +1M
    • find . -name “*.tmp” -exec rm {} 끝에 ;가 있으면 각각 명령어 실행 +가 있으면 찾은 파일 한번에 실행

  • which - path에 존재하는 파일을 검색

  • UNIX file mode

    • file mode bit: 유닉스 파일 권한을 나타내는 3 + 9 bit 체계
    • octal mode: 8진수로 표기되는 UNIX file mode
    • rwxrwxzxrwx: 3칸씩 각각의 owner, group, others 로 나눈다
    • rwxr-x— == 750
    • r == 4 readable
    • w == 2 writable
    • x == 1 executable
    • directory 인경우 400 인 경우 파일 내용 볼 수 있음
    • 파일생성시 기본값은 umask 022 를 뺀 755 가 됨
    • 권한 바꾸기 - chmod 664 testdir / rw-rw-r– r은 목록에서 파일을 볼수 있고, x는 파일에 access할 수 있음.
    • chmod -x hello / chmod +x hello / chmod u+x g=rw ..

  • file meta data

    • magic data: /usr/xhare/file/magic
    • mtime: modification time - file의 data가 변경된 시간
    • ctime: change time - file의 meta-data가 변경된 시간

  • stdio

    • standard input output
    • file channel: fild에 입출력하기 위한 통로, 표준화된 입출력 방식을 통하도록 하는 가상화 레이어
    • fd: 파일 서술자 file descrobtor, 파일 채널에 붙이는 유일한 식별자. 프로세스에 종속
    • 0: stdin - standard input 표준입력
    • 1: stdout - standard output 표준출력
    • 2: stderr - standard error 표준에러
    • 표준 입출력(0,1,2)는 모든 프로세스에 예약됨. 프로세스가 파일을 열면 fd는 3 부터 매겨짐.
    • 같은 파일을 여러번 열면 하나의 파일에 fd가 여러개 생성되며 싱크 이슈가 있을 수 있다.

pipe

  • 프로세스 사이에 통신으로 사용

  • IPC(inter-process communication)의 일종, shell에서도 사용

  • anonymous pipe - temporary 프로세스 종료시 사라짐

    • 프로세스틀의 직렬 연결
    • 명령행에서 vertical bar | 로 사용
    • A | B | C - A의 출력이 B로 들어가고 C로 또 연결됨 e.g. find . | wc -l find 의 출력이 wc의 입력과 연결된다.

  • named pipe - persistency

    • FIFO pipe 라고 부르기도 함
    • path + filename 이 있다. mkfifo 명령어

  • redirection

    • 방향재지정
    • A > B - A의 stdout 을 파일 B 로 연결 e.g. ls > filelist.txt strace ls 2> strace.txt
    • A < B - A의 stdin 을 파일 B 로 연결 e.g. ls < sort < names.txt
    • A >> B - 방향은 > 과 같고, 추가하는 모드

아카이브, 압축

  • 유닉스는 묶음과 압축이 분리되어있음
  • 아카이브 - 보관용 묶음 tar, cpio (BSD, SysV)
  • tar [ctxv] [f archive-file] files…
    • c: 아카이브
    • t: 아카이브 테스트
    • x: 아카이브로부터 파일을 풀어냄
    • v: 상세한 정보 출력 쓰지마
    • f archive-file 입출력할 아카이브 파일명
    • –exclude file: 제외할 file
  • 압축 - gzip, bzip2, xz, zstd, lz4 압축률 xz > bzip2, zstd > gzip > lz4
    • xz: 느리지만 텍스트 압축에 강함. 소스코드 압축할 때 씀.
    • zstd: 가장 좋음. 이거 써라. zip standard
    • tar c /etc/*.conf | gzip -c > etc.tar.gz - gzip -cd etc.tar.gz | tar x
    • bzip, xz, zstd 사용법 gzip 과 같음
    • verbose 옵션을 사용하지 마라. 성능에 영향을 줌.
    • tar cfa bak_data.tar.zst ./data ./exp -> tar c ./data ./exp | zstd -T0 > bak_data.tar.zst
    • tar xfa bak_data.tar.zst -> zstd -dcT0 > bak_data.tar.zst | tar x

link

  • ln - make links

  • inode - 파일의 메타 정보 및 관리용 객체

    • 파일은 고유의 inode를 1개 가지고 있다.
    • disk partition(or volume)내에서 유일한 식별자. 다른 볼륨이나 파티션에서는 의미 없음
    • file meta data - 시간 관련 정보, 사이즈, 소유권, 권한 등등

  • 하드 링크 - hard link - inode를 가리키는 링크

    • 같은 inode 를 가리키므로 동일 파티션 내에서만 생성 가능
    • regular file 만 가능 directory, device file 등은 hard link를 생성 할 수 없음. 실체를 가진 파일
    • ln

  • 심볼릭 링크 - symbolic link - 특정 파일의 위치를 가리키는 기능을 하는 링크(바로가기)

    • symlink 위치만 가리키므로 다른 파티션도 가능, 모든 종류의 file에 만들 수 있다.
    • 가리키는 대상의 UNIX file mode를 따라가므로 symlink 권한은 777이다. 크게 의미 없음
    • ln -s

  • readlink

    • symlink가 여러 단계로 가리키는 파일이 있을 수 있다. 이 canonical path를 따라가는 기능
    • readlink -f - 마지막 링크를 제외한 모든 링크가 존재할 떼 성공
    • readlink -e - 모든 링크가 존재할 때 성공

  • 사용 예

    • 매일 추천 음악 100에 해당하는 파일을 제공할 때, 매일 파일을 복사하지 않고 symlink가 가리키는 canonical path를 변경하며 사용한다.
    • 라이브러리 버전 관리도 유사하게 사용. /usr/lib 내부 확인해보자.

  • canonicalization

    • canonical이란 컴퓨팅 환경에서 실체를 가지는 standard, official의 의미를 가진다.
    • 예를 들어 상대경로를 절대 경로로 바꾸어 주는 과정, localhost 주소를 127.0.0.1 로 바꾸는 것 등을 canonicalization 한다고 표현한다.
    • symlink는 실체를 가리키는게 아니기 때문에 끝까지 따라가야 canonical path를 알수 있다.

process

  • ps - process status

    • pid: process ID
    • TTY: terminal ID
    • TIME: CPU 시간(누적 시간) = CPU를 점유했던 시간의 누적값. 현실 시간이 아님
    • CMD: command = 프로세스 이름. argv[0]를 의미

  • ps -e: 시스템의 모든 프로세스 조회

  • ps -a: ??

  • ps -f: full-format

    • UID: 해당 프로세스의 소유권자 ID. User ID
    • PPID: 부모 프로세스 ID (parent pid)
    • C: CPU 사용량 누적된 수치의 비율
    • STIME: 프로세스를 시작한 시간(시:분)

  • ps -l: long-format

    • F: 프로세스 플래그
    • S: 상태 코드(state code)
    • PRI: 실시간 우선순위(realtiem priority)
    • NI: 나이스 우선 순위(nice) -20 ~ +19 음수일수록 우선 순위가 높다. 기본값은 0
    • SZ: 사용되는 프로세스 코어 이미지의 메모리 크기 (KB)

  • ps axf -> BSD style / ps -e –forest -> UNIX SysV style

  • 명령어는 man ps으로 사용해 살펴보자.

  • process control

    • kill- 프로세스에 시그널을 send 하는 기능. 시그널에 따라 프로세스를 제어.
    • 시그널 종류는 kill -l 통해 확인.
      • SIGHUP: Hang Up
      • SIGINT: Interrupt
      • SIGQUIT: Quit <CTRL->. core 발생
      • SIGKILL: kill 강제로 죽임
      • SIGSEGV: Segment violation. 메모리 침범으로 이상 종료
      • SIGTERM: Terminate 종료 요청
      • SIGTSTP: Temporary Stop
    • kill 13011 - PID 13011 프로세스에 SIGTERM 시그널 보냄
    • kill -QUIT 12012 - SIGQUIT 시그널 보냄
    • kill -9 12013 - 9번 시그널(SIGKILL) 보냄 최후의 수단! SIGTERM을 기본적으로 시도

  • job control

    • fore-ground processs - 현재 session에서 제어 터미널(controlling terminal)을 가진 프로세스
    • back-ground process - 현재 session에서 제어 터미널(controlling terminal)을 잃어버린 프로세스
    • CTRL-z
      • SIGTSTP( signal - temporary stop) 시그널을 fore-ground 프로세스에 전달
      • 프로세스를 잠시 정지시킴 = 결과적으로 back-ground에 stpped 상태로 존재
    • session - 세션은 멀티유저 시스템에서 통신 객체(seat of remote)를 구별하기 위함.
      • 세션은 제어 터미널(Controlling terminal)을 가질 수 있다.
      • SID == PID 가 같은 프로세스를 Session Leader 라 부른다. 세션 리더로부터 파생된 자식 프로세스는 모두 같은 세션을 가진다.
      • 세션에 속한 controlling terminal을 소유할 수 있다는 의미 == fore-ground process
      • 로그아웃시 세션이 파괴되면서 세션에 속한 프로세스들은 모두 종료
    • controlling terminal
      • 사용자의 제어를 받는 터미널 장치
      • CUI에서 멀티 태스킹을 위한 제어 방법
      • 제어 터미널을 소유한 프로세스는 키보드 입력을 가진다. 이런 프로세스를 fore-ground process라 부른다.
      • 하나의 session에서 fore-ground process는 최대 1개
      • 제어 터미널 규격 TTY 필드 부분을 보면 두가지 방식
        • pts/#: UNIX98 Pseudo terminal system - SUSv2
        • tty#: console terminal
      • session에서 제어 터미널을 가지지 않는 경우 ps의 TTY 필드에 ? 로 나타난다.
    • process group
      • process group leader: processgroup ID == PID
      • 프로세스 그룹 리더는 시그널 전파 기능을 가짐.
      • kill 에서 음수의 PID 번호는 프로세스 그룹에 시그널을 전파하는 것.
    • deamon 이란??
      • Orphan process( and Session leader)
      • stdio를 모두 /dev/null로 리다이렉션을 걸고
      • control terminal 을 가지지 않는 프로세스. TTY 값 ?이다.
    • interrupt key
      • : SIGINT(Interrupt Signal) 시그널을 fore-ground 프로세스에 전달. 프로세스 그룹 리더가 받을 경우 그룹에 속한 모든 프로세스에 시그널 전파
      • <CTRL->: SIGQUIT(Quit Signal) 시그널을 fore-ground 프로세스에 전달. 동작 방식은 위와 같으나 core 발생
    • commands
      • jobs: stoped, back-ground process의 리스트 출력
      • fg %#: 지정한 프로세스를 fore-ground로
      • bg %#: 정지된 백그라운드 프로세스를 back-ground에서 running 상태로 변경
      • command &: command를 back-ground에서 running 상태로 실행

UNIX

  • Ken Thompson, Ritchie이 게임을 위해 UNIX를 만듬
  • UNIX가 성공하면서 DEC 운영체제가 밀려남
  • 당시 UNIX는 어셈블리어로 작성되어 새로운 CPU가 나오면 거의 새로 작성되어야 했다.
  • Dennis Ritchie는 Brian Kernighan의 도움으로 B 언어를 개량한 C 언어를 개발(1973)
  • UNIX가 C언어로 작성되면서 포팅이 쉬워짐
  • C언어의 부흥으로 어셈블리어의 몰락을 가져옴
  • UNIX는 분열하기 시작
  • BSD UNIX가 탄생 (1975년) 했으나 구심점이 없었다.
  • 기능을 각자 알아서 추가하다 보니 호환성이 사라짐
  • Sysv vs BSD 경쟁 - 명령어도 다름. 운영체제 버전이 다르면 호환이 안됨
  • POSIX가 있어도 더 복잡해진 상황
  • 빌게이츠의 제안으로 SUS - Single UNIX Specification 유닉스 단일 표준안

C Programming Language

  • The C Programming Language (TCPL, K&R)
  • 고급언어이면서도 어셈블리어에 버금가는 성능
  • 낮은 추상화 지원 stdio, File 등
  • 저수준의 하드웨어 조작이 가능
  • 어셈블리어보다 쉬운 언어적 특성
  • ANSI-C(C89): 1989년도 표준
  • C99 - 1999년도 표준
  • C11 : 2011년도 표준

POSIX

  • Portable Oprating system Interface
  • 1988년 POSIX 초안 발표
  • 전기전자공학회(IEEE)의 주도로 system call 표준화 진행
  • POSIX는 API의 문법과 작동에 대한 의미(semantic)만을 담고 있음. 느슨하게 작성됨.
  • 정부 조달의 최소기준

OSF

  • Open Software Foundation 등장
  • 대기업의 반독점을 목적으로 설립된 비영리 단체
  • 1990년 초반 BSD, SVR4, OSF/1 등으로 파편화

X/Open
