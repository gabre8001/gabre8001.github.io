SvelteKit RBAC

네, SvelteKit에서 RBAC(Role-Based Access Control)를 구현하는 것은 매우 중요한 주제이다. 사이드 메뉴, URL, API 이 세 가지 관점에서 접근을 제어하는 방법을 단계별로, 그리고 통합적인 아키텍처 관점에서 설명한다.

핵심은 SvelteKit의 hooks.server.ts 와 +layout.server.ts 를 사용하여 인증 상태와 역할을 중앙에서 관리하고, 이를 UI와 API 전반에 전파하는 것이다.

🏛️ 기본 아키텍처: 인증 및 세션 관리

RBAC의 가장 기본은 “현재 사용자가 누구이며, 그의 역할(Role)은 무엇인가?” 를 아는 것입니다. 이 과정은 보통 로그인 시 시작되며, 서버는 사용자의 역할을 포함한 세션 정보를 생성합니다.

1단계: hooks.server.ts에서 세션 처리하기 (중앙 허브)

hooks.server.ts는 모든 서버 측 요청(페이지 로드, API 호출)이 처리되기 전에 실행되는 "미들웨어"입니다. 여기서 사용자의 세션을 확인하고, 사용자 정보를 event.locals에 저장하여 요청 처리 과정 내내 사용할 수 있도록 합니다.

src/hooks.server.ts

src/app.d.ts
event.locals.user에 타입을 추가하여 TypeScript의 이점을 누릴 수 있도록 설정합니다.

2단계: +layout.server.ts에서 UI에 사용자 정보 전달하기

루트 레이아웃(src/routes/+layout.server.ts)의 load 함수는 모든 페이지가 로드될 때 실행됩니다. 여기서 event.locals의 사용자 정보를 data 객체에 담아 프론트엔드로 전달합니다.

src/routes/+layout.server.ts

이제 모든 Svelte 컴포넌트에서 data.user를 통해 현재 로그인된 사용자의 정보(역할 포함)에 접근할 수 있습니다.

🔐 3가지 관점에서의 접근 제어 구현

위의 기본 아키텍처를 바탕으로, 요청하신 세 가지 접근 제어를 구현해 보겠습니다.

1. Side Menu (UI 관점) 접근 제어

가장 간단한 부분입니다. 루트 레이아웃이나 사이드 메뉴 컴포넌트에서 data.user의 role을 확인하여 조건부로 메뉴를 렌더링합니다.

src/lib/components/SideMenu.svelte (예시 컴포넌트)

장점: 사용자는 자신의 권한에 맞는 메뉴만 보게 되므로 사용자 경험이 향상됩니다.
한계: 이것만으로는 보안이 충분하지 않습니다. 주소창에 URL을 직접 입력하는 것을 막을 수는 없습니다.

2. URL (라우팅 관점) 접근 제어

사용자가 권한이 없는 페이지의 URL로 직접 접근하는 것을 막는 ‘라우트 가드(Route Guard)’ 기능입니다. 이 로직은 중앙 허브인 **hooks.server.ts**에 구현하는 것이 가장 효율적입니다.

src/hooks.server.ts (확장된 버전)

장점: URL 직접 접근을 서버 차원에서 차단하므로 강력한 보안을 제공합니다.
동작: 사용자가 /admin 페이지를 요청하면, handle 함수가 먼저 실행되어 사용자의 역할이 admin이 아니면 /unauthorized 페이지로 즉시 리다이렉트시킵니다.

3. API 호출 제어 (백엔드 관점)

가장 중요한 보안 계층입니다. 프론트엔드 UI가 어떻게 조작되더라도, 최종적으로 데이터를 처리하는 API 서버는 스스로를 보호해야 합니다. 이 로직은 각 API 엔드포인트(+server.ts) 파일의 시작 부분에 구현합니다.

src/routes/api/users/delete/+server.ts (사용자 삭제 API 예시)

장점: 최종 데이터 관문에서 권한을 검사하므로 가장 안전합니다. curl이나 다른 도구를 이용한 직접적인 API 호출 시도도 완벽하게 차단합니다.
동작: API 요청이 들어오면, hooks를 통해 locals.user에 사용자 정보가 담깁니다. API 핸들러는 이 locals.user.role을 확인하여 admin이 아니면 즉시 403 Forbidden 에러를 반환하고 로직을 중단합니다.

🎨 종합적인 흐름

  1. 사용자 로그인: 성공 시, role이 포함된 세션 토큰이 쿠키에 저장됩니다.
  2. 모든 요청: hooks.server.ts가 쿠키를 읽어 event.locals.user에 사용자 정보를 설정합니다.
  3. URL 접근: hooks.server.ts가 URL 경로와 locals.user.role을 비교하여 권한이 없으면 리다이렉트시킵니다.
  4. 페이지 로드: +layout.server.ts가 locals.user를 프론트엔드로 전달합니다.
  5. UI 렌더링: SideMenu.svelte가 전달받은 data.user.role에 따라 조건부로 메뉴를 보여줍니다.
  6. API 호출: +server.ts 파일이 locals.user.role을 확인하여 권한이 없으면 403 에러를 반환하고, 있으면 로직을 실행합니다.

이 아키텍처는 SvelteKit의 기능을 최대한 활용하여 세 가지 관점에서 일관되고 안전하게 RBAC를 구현하는 강력한 패턴입니다.
